Project Development and Publishing Guidelines for PunktNameChanger

## 1. Code Standards
   - All code comments must be in English.
   - Keep the browser app client-side only unless a task explicitly changes that architecture.
   - Keep the single-file and split-file implementations behaviorally aligned.
   - Maintain clean, readable code structure.
   - Avoid unrelated refactors during field-fix updates.

## 2. Interface Requirements
   - Keep the UI touch-friendly for smartphone field usage.
   - Preserve readable labels, validation messages, and logging.
   - Test mobile browser usability after layout changes.
   - Keep all user-facing text in English.

## 3. Renaming Rules
   - Support dot-format point IDs only: G01.001..G10.998 and P01.001..P10.998.
   - Preserve source file formatting and fixed-width alignment.
   - Validate coordinates against the selected master before renaming.
   - Respect the hard QTY limit for every pattern run.
   - Keep MQ numbering tied to the original source point pair index, not just the count of encountered rows.
   - Partial measurements with skipped point ranges must keep their natural MQ positions.

## 4. Documentation Standards
   - README.md, Mission.md, AGENTS.md, rules.txt, Function.txt, and VALIDATION.md must be in English.
   - Document supported formats and safety behavior.
   - Update documentation when renaming behavior changes.

## 5. Repository Management
   - Use .gitignore to exclude generated local files and development-only artifacts.
   - Publish essential project files, documentation, validation tests, and license files.
   - After each functional update, run validation tests, commit the changes, and push to GitHub.
   - Do not leave completed project changes only in the local working tree unless the user explicitly asks to pause before publishing.
   - Keep commit messages descriptive and in English.

## 6. Quality Assurance
   - Run `python tests/run_validation.py` after functional changes.
   - Add regression cases for every field bug that changes renaming logic.
   - Test `.imes` and `.ipkt` fixed-width replacements when changing pipe-format logic.
   - Test `.iroh` and `.lqp` guards when changing cross-format renaming behavior.

## 7. Project Structure
   ```
   index_singlefile_mobile.html  # All-in-one mobile app
   index.html                    # Split-version HTML shell
   css/style.css                 # Mobile-first styles
   js/utils.js                   # Shared helpers and logging
   js/parsers.js                 # Coordinate map building
   js/renamer.js                 # Renaming engine
   js/main.js                    # UI orchestration and export
   tests/run_validation.py       # Regression validation suite
   AGENTS.md                     # Agent instructions
   rules.txt                     # Development and publishing rules
   LICENSE                       # MIT license
   VALIDATION.md                 # Validation notes
   Function.txt                  # Behavior notes and source-of-truth guidance
   ```
